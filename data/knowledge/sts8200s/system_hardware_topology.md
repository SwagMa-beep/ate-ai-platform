# STS8200S system hardware topology

适用场景：
- 理解 STS8200S 测试系统从 PC 到测试资源的硬件链路
- 评审 ResourceMap、治具连接、资源冲突时判断问题落在哪一层
- 排查“软件配置正确但资源不可用”这类问题

系统硬件主链路：
- PC 机负责工程、程序、界面和结果管理
- 总线接口卡负责把主机控制指令送到测试主机/插件箱
- 测试主机与插件箱承载资源模块、电源单元、总线板和机内电缆
- 资源转接板把平台内部资源引出到适配器和 DUT
- 适配器再把具体资源分配到被测芯片各引脚

核心能力 / 限制：
- 软件里看到的模块并不等于当前机台实际安装的模块
- 同一个测试方案能否落地，既取决于程序，也取决于插件箱内模块数量和插位类型
- 资源转接板和适配器是很多“连不上、切不到、量不到”的根因所在
- 机内电缆、总线板和转接板一旦链路不完整，程序层面的 HardWareCfg 也无法弥补

典型工程规则：
1. 先确认系统版本、插件箱和已安装模块清单，再评审 TestPlan 是否可实现
2. 先确认资源转接板和适配器是否支持目标信号路径，再写 ResourceMap
3. 多工位规划时，要区分“软件 site 概念”和“真实资源数量”
4. 软件配置通过不代表硬件一定就绪，最好同时结合配置检查和硬件链路检查

典型排查顺序：
1. 确认 PC / 总线接口卡 / 主机通信正常
2. 确认插件箱中对应模块实际存在且插位正确
3. 确认资源转接板与适配器版本匹配
4. 确认适配器路径上继电器、连接器、线缆没有断路或错接
5. 再检查 HardWareCfg、Site 绑定、模块编号和 ResourceMap

工程建议：
- 在项目立项或竞赛准备阶段就建立“机台资源配置清单”
- TestPlan 审查时，把每个关键测试项需要的模块类型和数量写出来
- 适配器评审时，不只看引脚连通，也要看资源是从哪块转接板引出的
- 调试时如果现象像“资源完全不可用”，优先怀疑硬件链路而不是算法本身

常见风险 / 注意事项：
- 把机台标准配置当成当前机台实际配置
- 忽略资源转接板或适配器版本差异
- 只看程序分配，不看模块安装数量和物理路径
- 多工位数量按理想值规划，实际却被稀缺模块卡住
