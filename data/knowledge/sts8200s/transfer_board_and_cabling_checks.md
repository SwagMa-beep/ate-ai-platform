# STS8200S transfer board and cabling checks

适用场景：
- 排查“程序配置看起来正确，但资源还是不可用”的问题
- 评审资源转接板、机内电缆、适配器连接链路
- 多工位或混合资源项目上线前做硬件链路核查

核心能力 / 限制：
- STS8200S 的资源不是直接从模块跳到 DUT，中间通常还经过机内电缆、资源转接板、类别板和适配器
- 资源转接板负责把机柜内部资源灵活引出到不同类别板和适配器
- 一旦转接板版本不匹配、线缆接错、插位错误，程序层的 HardWareCfg 无法补救

典型硬件链路：
- 插件箱模块
- 总线板 / 电源总线
- 机内电缆
- 资源转接板
- 类别板 / 适配器
- DUT 引脚

典型检查规则：
1. 先确认模块实际安装位置
   - 插件箱插位、模块型号、资源数量必须和方案一致
2. 再确认机内电缆连接
   - FOVI、DIO、MPOW、多工位资源板相关线缆要逐一核对标签
3. 再确认资源转接板版本与接法
   - 同一套程序不能默认兼容所有转接板和适配器版本
4. 最后再检查 ResourceMap 和 HardWareCfg
   - 程序配置必须建立在真实可达的物理路径上

工程建议：
- 在 ResourceMap 评审表中增加“转接板/线缆路径确认”一栏
- 多工位项目中，把共享资源和隔离资源在转接板层就标出来
- 调试初期可先用最短路径、最少继电器切换确认基础资源可达
- 若出现资源偶发不可用，优先怀疑接插件、线缆接触或转接板路径，而不是先改算法

常见风险 / 注意事项：
- 模块装了，但线缆没接到正确转接板
- 转接板版本与适配器版本不一致
- 多工位共板时，某个工位路径被遗漏或串到错误资源
- 程序配置正确，却被物理链路断点拖住
