# Full ATE Development Flow

## 目标

`full_ate_development` 是跨模块的 ATE 开发 Agent Flow。

它的目标不是替代模块一、模块二、模块三现有页面，而是在统一 `run / step / artifact` 模型上，提供一条从 Datasheet 到测试工程交付物的全链路自动化流程。

## 当前第一版范围

当前第一版已经打通以下步骤：

1. `输入解析`
2. `TestPlan 提取`
3. `参数校验`
4. `资源映射`
5. `RAG 检索`
6. `测试规划`
7. `代码装配`
8. `静态校验`
9. `编译预检`
10. `工程复核`
11. `工程打包`

这条 flow 已能通过统一 `AgentController` 执行，并把中间结果记录进 run 和 artifacts。

## 步骤说明

### 1. 输入解析

职责：

- 接收 `goal`
- 解析 `file_id` 或 `pdf_path`
- 确认目标平台
- 为后续步骤准备统一输入

### 2. TestPlan 提取

职责：

- 调用模块一提取能力
- 从 Datasheet 抽取参数、引脚、测试场景
- 沉淀结构化提取结果

### 3. 参数校验

职责：

- 检查提取结果是否足以支撑后续流程
- 标记是否需要人工复核

当前重点校验：

- 是否提取到引脚定义
- 是否识别到电源 / 地
- 是否识别到结构化参数

### 4. 资源映射

职责：

- 调用模块二资源映射能力
- 生成 STS8200S 资源映射结果

### 5. RAG 检索

职责：

- 根据目标和上下文检索 STS8200S 知识
- 为后续代码生成提供附加上下文

当前策略：

- 如果 RAG 未就绪或无有效命中，不中断流程
- 回退到企业知识库与模板生成

### 6. 测试规划

职责：

- 根据提取结果、企业知识库和约束生成代码规划
- 明确需要的测试项、向量依赖和风险点

### 7. 代码装配

职责：

- 复用现有模块三代码生成能力
- 输出 C++ 测试代码骨架

### 8. 静态校验

职责：

- 进行静态规则检查

### 9. 编译预检

职责：

- 进行结构化编译预检
- 当前仍以本地预检为主，不等同于真实 STS 工程编译

### 10. 工程复核

职责：

- 汇总参数校验、RAG、静态校验、编译预检、VEC/DIO/PGS 依赖
- 输出人工复核结论

当前策略：

- 自动生成结果永远不直接标记为“可直接上机”
- 必须给出人工复核提醒

### 11. 工程打包

职责：

- 在具备 `file_id` 且开启导出时，生成工程包
- 输出工程目录、代码、计划文件和 starter 产物

## 当前 fallback 策略

### RAG fallback

- RAG 未就绪：继续流程，写入 warning
- RAG 无命中：继续流程，提示人工重点复核平台 API 使用

### 代码生成 fallback

- 优先依赖现有企业知识库和模板能力
- 不把 LLM 输出作为无约束的唯一最终结果

### 编译预检 fallback

- 若没有真实编译环境，保留结构化预检结果
- 明确提示需要在真实工程环境复核

## 当前边界

当前第一版 full flow 仍属于“可运行骨架”，不是最终完成态：

- 已具备统一执行链路
- 已具备 review summary
- 已具备 artifacts 记录

但后续仍可继续增强：

- 更强的条件路由
- 更完整的 retry / fallback 策略
- 更细的 review 节点
- 更深的模块二导出联动
- 更完整的工程交付物索引与下载

## 使用方式

通过：

```text
POST /api/v1/agent-runs
```

传入：

- `flow_name=full_ate_development`
- `goal`
- `file_id` 或 `pdf_path`
- 可选的代码生成与导出参数

系统会返回完整 run，并可在运行中心查看：

- step 时间线
- artifacts 摘要
- review summary
