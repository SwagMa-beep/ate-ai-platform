# 阶段二完成说明：模块一、二、三统一 Run 模型

## 目标

阶段二的目标不是继续堆新功能，而是把平台三条主链收进同一套运行模型，让系统从“各模块各自运行”升级到“统一可追踪、可回看、可排错”的平台形态。

本阶段重点完成：

- 模块一提取流程接入统一 `controller / run / step / artifact`
- 模块二资源映射流程接入统一 `controller / run / step / artifact`
- 模块三代码生成流程完成一阶段 agent 化后的统一运行展示
- 前端三个主页面都能感知本次运行摘要
- 新增统一的“运行中心”查看运行历史和中间产物

---

## 本阶段已完成内容

### 1. 模块一纳入统一运行模型

已完成内容：

- 同步提取接口改为通过统一 controller 执行
- 异步提取完成后会补录统一 run 记录
- 运行结果会保存 step、artifact 和最终结果摘要
- 提取页增加“本次运行摘要”
- 运行中心可查看模块一提取记录

当前模块一步骤语义：

- `输入解析`
- `TestPlan 提取`

当前模块一产物语义：

- `原始 PDF`
- `提取结果`

### 2. 模块二纳入统一运行模型

已完成内容：

- 资源映射主流程改为通过统一 controller 执行
- 映射结果、导出结果与 run 记录关联
- 资源页增加“本次运行摘要”
- 运行中心可查看模块二资源映射记录

当前模块二步骤语义：

- `映射输入解析`
- `资源映射`

当前模块二产物语义：

- `映射输入`
- `资源映射结果`

### 3. 模块三完成一阶段 agent 化并纳入统一运行模型

已完成内容：

- 代码生成从路由手工串联 service，升级为 controller 编排 flow
- 已拆分标准步骤：
  - `测试规划`
  - `代码装配`
  - `静态校验`
  - `编译预检`
  - `工程打包`
- 每次生成都会记录 run、step、artifact
- 代码实验室支持显示本次运行摘要
- 运行中心可查看模块三完整运行记录和中间产物

当前模块三典型产物：

- `生成计划`
- `代码结果`
- `静态分析`
- `编译预检结果`
- `工程包`

### 4. 新增统一“运行中心”

已完成内容：

- 增加独立页面查看最近运行记录
- 支持统一查看模块一、模块二、模块三 run
- 支持查看：
  - 当前状态
  - 已完成阶段数
  - 需关注阶段数
  - 产物数量
  - 阶段时间线
  - 产物摘要

页面定位：

- 运行中心用于排错、回看流程、理解中间产物
- 它不是主操作页，不替代提取器、资源映射、代码实验室

### 5. 前端术语和状态展示统一

已完成内容：

- 抽出共享运行展示工具
- 统一状态词：
  - `已完成`
  - `已阻断`
  - `执行中`
  - `待执行`
- 统一流程名、步骤名、产物名映射
- 三个主页面都使用同一套 run 展示语义

---

## 本阶段产出结果

经过阶段二改造，平台当前已经具备下面这套统一能力：

- 一次模块操作可以沉淀为一条 run
- 一条 run 可以分解为多个 step
- 每个 step 都可挂接 artifact
- 三条主链都能进入统一运行历史
- 用户既可以在主页面完成操作，也可以去运行中心排错和回看过程

这意味着平台已经从“分散的页面功能”进化为“带统一运行模型的工程平台”。

---

## 当前边界

阶段二完成后，平台已经完成了主链统一运行模型，但仍有明确边界：

### 1. 还不是全平台完全 agent 化

当前更准确的说法是：

- 模块三完成了一阶段 agent 化
- 模块一、模块二完成了统一 run 模型接入

也就是说，平台已经具备统一 controller/run 的主干，但还没有进入“全平台强 agent 协同”阶段。

### 2. 运行中心仍偏排错视图

运行中心现在更适合：

- 查看历史
- 理解流程
- 定位问题

它还不是最终意义上的“平台运行指挥中心”。

### 3. 模块三虽然已经 run-first 化，但交互还可继续收紧

当前模块三已经体现“先运行、后结果”的思路，但仍然需要继续优化信息密度和页面层级，让主操作区更轻、更像产品页。

---

## 阶段二的工程价值

从工程角度看，本阶段最重要的价值不是多加了一个页面，而是建立了统一运行模型：

- 后端不再只是零散 service 调用
- 前端不再只拿最终结果黑盒展示
- 系统开始具备“过程可见、结果可追溯、问题可定位”的平台特征

这会直接支撑下一阶段继续扩展：

- 更完整的 agent controller 能力
- 更强的条件分支与 review/fallback
- 更多模块纳入统一运行治理

---

## 下一阶段建议

### 1. 收尾与清理

优先继续完成：

- 清理前端残留历史文案和乱码
- 统一运行摘要卡片样式
- 收敛重复展示逻辑

### 2. 强化模块三 run-first 体验

重点继续优化：

- 让代码实验室更突出“运行主线”
- 进一步降低非必要信息密度
- 把详细排错更多收敛到运行中心

### 3. 进入更强的 agent controller 阶段

下一阶段可以考虑继续补：

- 条件分支
- review 节点
- fallback agent
- step 级 retry
- 更细的运行指标

---

## 一句话结论

阶段二已经完成了平台主链的统一运行模型建设：

- 模块一、模块二、模块三都已进入同一套 `run / step / artifact` 体系
- 前端主页面和运行中心都已能感知运行状态

这标志着项目已经从“功能型 workflow 平台”进一步升级为“具备统一运行主干的工程平台”。
