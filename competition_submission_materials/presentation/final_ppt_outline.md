## 第 3 页：项目背景与痛点

### 页面内容

建议放 ATE 测试开发流程图：

Datasheet 阅读 -> 参数整理 -> TestPlan 编写 -> DUT 资源映射 -> 测试代码开发 -> 调试验证 -> 良率诊断

痛点：

- Datasheet 内容复杂，人工整理耗时长。
- 参数、单位和测试条件格式不统一。
- DUT 资源映射依赖工程经验。
- 测试代码首版容易出现低级错误。
- 诊断数据和开发流程割裂。

### 讲稿

ATE 测试开发是芯片工程验证和量产测试中的关键环节。传统流程中，工程师需要先阅读 Datasheet，整理电气参数、引脚定义和测试条件，再编写 TestPlan、分配 DUT 资源、编写测试代码，最后进行调试和诊断。

这个流程存在几个明显问题：第一，Datasheet 表格和图形多，人工整理耗时；第二，不同厂商的参数格式差异较大；第三，资源映射高度依赖工程经验；第四，测试代码初稿容易出现通道、量程和条件设置错误；第五，测试开发和良率诊断工具通常是割裂的。因此我们希望用大模型和工程规则构建一个智能辅助平台，降低重复劳动和初版错误。

## 第 4 页：项目目标

### 页面内容

目标一句话：

构建从 Datasheet 到 TestPlan、资源映射、测试代码和诊断分析的智能 ATE 测试开发辅助闭环。

核心目标：

- 自动解析 Datasheet PDF。
- 自动生成 TestPlan JSON/Excel。
- 根据 STS8200S 平台规则进行参数校验。
- 生成 DUT 资源映射、PGS、BOM 和 SVG 示意图。
- 生成 STS8200S C++ 测试代码骨架。
- 提供轻量化良率诊断展示。
- 打包为 Windows 桌面应用，无需用户安装 Python。

### 讲稿

本项目的目标不是单独做一个 PDF 解析工具，而是做一个面向 ATE 测试开发流程的辅助平台。系统以 Datasheet 为输入，以 TestPlan、资源映射、测试代码和诊断结果为输出，形成一个较完整的工程闭环。

当前系统已经实现 Windows 桌面端交付，用户上传 Datasheet 后，可以完成参数提取、TestPlan 导出、资源映射生成、代码骨架生成和诊断展示。这个目标也对应了赛题要求中的识别准确率、DUT 辅助设计、代码生成、轻量化运行和效率提升五个方向。

## 第 5 页：系统总体架构

### 页面内容

建议放系统架构图，包含：

- 前端交互层：React + TypeScript + Electron。
- 后端服务层：FastAPI。
- AI 能力层：DeepSeek API + RAG。
- 工程规则层：STS8200S 规则、本地兜底规则、静态校验。
- 文件输出层：TestPlan、资源映射、BOM、SVG、C++ 代码。
- 部署层：PyInstaller 后端打包 + Windows 安装包。

### 讲稿

系统采用分层架构。前端使用 React 和 Electron，负责上传、展示和下载；后端使用 FastAPI，负责 PDF 解析、模型调用、规则校验、文件生成和诊断分析。

AI 能力层主要调用 DeepSeek API，并结合 RAG 知识增强实现代码生成。工程规则层是我们项目比较关键的部分，它把 STS8200S 平台约束、本地参数规则和代码静态校验结合起来，避免系统完全依赖大模型输出。最终系统通过 PyInstaller 和 Electron 打包为 Windows 桌面应用，用户无需安装 Python 就可以运行。

## 第 6 页：核心流程

### 页面内容

建议放流程图：

PDF 上传 -> 页面过滤 -> 文本/表格解析 -> LLM 结构化提取 -> 本地规则兜底 -> STS8200S 校验 -> TestPlan 生成 -> 资源映射 -> 代码生成 -> 诊断展示

### 讲稿

系统的核心流程分为十步。用户首先上传 Datasheet PDF，后端解析页面文本和表格内容。为了提升速度，我们会先过滤掉封面、目录、修订记录等低价值页面，只保留电气特性、绝对最大额定值、推荐工作条件、引脚描述等关键页面。

随后系统调用大模型进行结构化提取，并通过本地规则进行补充和清洗。提取结果会进入 STS8200S 平台规则校验，生成 TestPlan 文件。后续模块会继续读取 TestPlan，生成资源映射、PGS、BOM、SVG 和测试代码骨架，最后在诊断页面展示良率和故障分析结果。

## 第 7 页：关键技术一：Datasheet 智能解析

### 页面内容

展示内容：

- PDF 文本提取。
- 表格内容融合。
- 关键页面过滤。
- LLM JSON 结构化输出。
- 单位归一化和字段标准化。
- 本地规则兜底。

可以放一张提取器界面截图。

### 讲稿

第一个关键技术是 Datasheet 智能解析。Datasheet 中的信息通常分散在不同章节，有些是表格，有些是文字说明，也有些是图形 pinout。我们首先对 PDF 进行文本和表格解析，再通过页面过滤减少无效输入。

在模型调用阶段，我们通过 Prompt 约束模型输出统一 JSON 结构，包括参数名称、符号、最小值、典型值、最大值、单位、测试条件和参数分类。模型输出后，后端继续做字段标准化、单位归一化、重复项合并和异常值过滤。对于常见参数和部分已知 pinout，还加入本地规则兜底，提高结果稳定性。

## 第 8 页：关键技术二：STS8200S 工程规则校验

### 页面内容

展示规则类型：

- 电压范围校验。
- 电流范围校验。
- DIO 通道约束。
- CBIT/TMU/VI 等资源提示。
- 缺失测试条件告警。
- 量程建议。

### 讲稿

第二个关键技术是工程规则校验。单纯让大模型从 Datasheet 中抽取文本，并不能保证结果适合真实 ATE 平台。因此我们把 STS8200S 平台约束加入系统，对提取到的参数进行二次校验。

例如，系统会检查电压和电流范围是否合理，DIO 通道是否满足需求，是否存在缺失测试条件，是否需要特殊量程，并在 TestPlan 中给出 warning 和建议。这样输出结果就不只是文本摘要，而是更接近测试工程师可复核、可使用的工程文件。

## 第 9 页：关键技术三：TestPlan 自动生成

### 页面内容

展示输出：

- TestPlan JSON。
- TestPlan Excel。
- A/B/C 类参数分类。
- DC/AC/LDO 测试项统计。
- 风险提示和量程建议。

建议放 Excel 或前端统计卡片截图。

### 讲稿

在 TestPlan 生成模块中，系统会把模型抽取结果和规则校验结果统一转换为 JSON 和 Excel 两种格式。JSON 用于系统内部流转，Excel 则方便测试工程师复核和二次编辑。

系统会按照 A 类必测参数、B 类推荐参数、C 类可选参数进行分类，同时统计 DC、AC、LDO 等测试项数量，并写入 STS8200S 兼容性提示和风险信息。当前这部分完成度较高，是整个系统的核心输出之一。

## 第 10 页：关键技术四：DUT 资源映射与辅助设计

### 页面内容

展示生成文件：

- BOM 清单。
- SVG 辅助示意图。

### 讲稿

资源映射模块会读取前面生成的 TestPlan 和引脚定义，根据引脚方向、芯片类型和测试项需求，把 DUT 引脚映射到 STS8200S 平台资源。

系统可以生成资源映射表、PGS 配置、BOM 清单和 SVG 辅助示意图。对于自动识别不完整的引脚，系统也提供 PinMapping 模板，允许工程师补充。当前它已经可以处理基础数字芯片、LDO 和常见小规模芯片场景，后续会继续扩展复杂多电源、多站点和模拟量测规则。
