# 基于语言大模型的智能 ATE 测试开发辅助平台结题技术报告

## 摘要
本项目面向集成电路 ATE 测试开发场景，设计并实现了一套基于语言大模型的智能测试开发辅助平台。系统以芯片 `Datasheet` 为输入，围绕 `Datasheet -> TestPlan -> 资源映射 -> 测试代码 -> 工程复核 -> 工程包` 主链路，提供 PDF 解析、参数提取、引脚识别、`STS8200S` 资源映射、RAG 增强代码生成、良率诊断展示、统一运行中心和工程师助手能力。平台采用 `React + Electron + FastAPI` 的桌面化架构，后端结合大模型能力、工程规则和知识库增强，输出可供测试工程师继续复核和使用的开发交付物。

截至结题阶段，项目已完成主框架开发、核心链路联调、前后端一体化展示、工程师助手迁移和阶段性指标验证，具备较完整的演示和阶段性交付能力。项目当前定位为 **ATE 开发辅助与交付平台**，已能够提升资料整理、资源规划和代码初稿生成效率，但尚未实现批准后自动驱动 ATE 真机执行测试。

## 1. 项目背景与研究意义

### 1.1 项目背景
ATE 测试开发通常需要测试工程师完成以下工作：

- 阅读和整理 `Datasheet`
- 提取电气参数、引脚定义和测试条件
- 编写 `TestPlan`
- 完成 `DUT` 与 `STS8200S` 平台资源映射
- 编写测试程序初稿
- 进行静态检查、编译预检、调试与复核
- 在后续量产中结合波形和失效信息进行诊断

这一流程存在明显痛点：

- `Datasheet` 表格、图表和跨页内容复杂，人工整理耗时长
- 不同芯片厂商格式差异大，参数难以标准化
- 资源映射和平台配置高度依赖工程经验
- 测试代码初稿重复性高，且容易出现通道、量程、条件配置错误
- 开发过程与后续诊断工具割裂，缺少统一平台

### 1.2 研究意义
本项目将语言大模型的文档理解能力、知识增强能力与 `STS8200S` 工程规则相结合，探索将 AI 引入 ATE 测试开发流程的可行路径。项目的意义主要体现在：

- 降低测试工程师重复整理文档和搭建初版代码的工作量
- 提升参数提取、资源映射和测试代码初稿生成效率
- 通过工程规则和复核机制降低首版错误率
- 为后续真实机台执行闭环、客户级部署和协同开发提供平台基础

## 2. 课题目标与指标

本项目对应的课程指标如下：

1. 大模型对 `Datasheet` 中复杂表格和图表的识别准确率不低于 `85%`
2. `DUT` 原理图辅助设计建议准确率不低于 `75%`
3. 测试程序辅助编程，自动生成代码符合测试要求，具备可实现性
4. 工控机上运行的轻量化 AI 模型在不影响机台运行前提下完成实时诊断
5. 相比传统开发模式，测试程序开发周期缩短不低于 `50%`，代码首检错误率降低 `70%` 以上

本项目的技术目标与实现策略如下：

| 指标要求 | 对应实现目标 |
| --- | --- |
| `Datasheet` 识别准确率 ≥ 85% | 构建 PDF 解析、页面过滤、结构化提取、本地规则兜底与结果标准化链路 |
| 辅助设计建议准确率 ≥ 75% | 基于引脚定义和 `STS8200S` 资源规则自动生成资源映射、`PGS`、`BOM` 和 `SVG` |
| 自动生成可实现测试代码 | 基于模板、RAG、工程规则和静态校验生成可复核的 C++ 测试代码初稿 |
| 轻量化运行并完成诊断 | 基于 `Electron` 桌面端和本地后端构建轻量化运行环境，并提供诊断展示 |
| 周期缩短与错误率下降 | 串联提取、映射、代码生成和复核流程，形成统一辅助闭环 |

## 3. 系统总体架构

### 3.1 总体设计
系统采用“桌面端前后端一体化 + AI 服务增强 + 工程规则约束”的总体架构，核心由以下部分组成：

- 前端展示层：`React + TypeScript + Vite`
- 桌面封装层：`Electron`
- 后端服务层：`FastAPI`
- AI 能力层：文本大模型、RAG 检索增强、工程师助手
- 工程规则层：`STS8200S` 平台规则、静态检查、资源映射规则、复核规则
- 文件与交付层：`TestPlan`、资源映射、`PGS`、`BOM`、`SVG`、代码文件、工程包

### 3.2 当前前端结构
当前主前端已形成统一工作台式结构，主要页面包括：

- `ATE Agent 工作台`
- `工程师助手`
- `Agent 运行中心`
- `Datasheet / TestPlan`
- `STS8200S 资源映射`
- `RAG 测试代码生成`
- `良率诊断`
- `知识库管理`
- `设置`

其中：

- `ATE Agent 工作台` 用于发起完整 `full_ate_development` 流程并查看执行轨迹
- `Agent 运行中心` 用于查看历史 `run`、产物和工程复核信息
- `工程师助手` 用于读取工作区上下文、知识片段和最近运行记录，提供问答与解释能力

### 3.3 当前后端结构
主后端统一挂载于 `backend/app/main.py`，核心接口包括：

- `/api/v1/testplan/*`
- `/api/v1/resource-map/*`
- `/api/v1/codegen/*`
- `/api/v1/diagnosis/*`
- `/api/v1/rag/*`
- `/api/v1/agent-runs/*`
- `/api/v1/chat/*`
- `/api/v1/workspace-memory/*`

统一运行模型以 `run / step / artifact` 为核心：

- `AgentController` 负责流程编排
- `RunStore` 负责运行记录与产物持久化
- `full_ate_development_flow` 负责完整开发主链
- `post_review_delivery_flow` 负责批准后的交付整理
- `post_review_revision_flow` 负责打回后的修订流转

## 4. 关键技术实现

### 4.1 PDF 解析与 TestPlan 提取
系统的 `Datasheet` 提取链路由以下环节组成：

1. 上传 PDF 文件
2. 解析页面文本与表格
3. 页面过滤与有效页筛选
4. 大模型结构化提取参数、测试项和引脚定义
5. 本地规则兜底与标准化
6. 输出 `TestPlan JSON` 与 `Excel`

当前版本已加入 `OCR fallback` 能力，用于文本层稀疏页面的兜底处理，但尚未将 OCR 作为全量主链启用。这样做的原因是文本型 PDF 直接解析更快、更稳定，而 OCR 适合作为扫描件或疑难页补充。

### 4.2 RAG 检索增强
RAG 主要服务于代码生成和工程解释环节。当前实现具备以下特点：

- 内置 `STS8200S` 平台知识片段
- 支持 TF-IDF 降级检索
- 支持企业知识资产扩展
- 对 `goal / chip / 测试项 / 平台 API` 做增强查询词构造
- 当 RAG 未命中时不中断流程，而是自动降级到模板与企业知识模式

RAG 当前主要用于：

- 提供平台 API 和开发规范上下文
- 提高代码生成与解释的一致性
- 为工程师助手提供本地知识片段支撑

### 4.3 资源映射与辅助设计
系统读取引脚定义和芯片参数后，按照 `STS8200S` 资源规则进行自动映射，输出：

- `resource_map.xlsx`
- `PGS` 配置
- `BOM`
- `schematic.svg`

映射过程中会对多电源、双工位、输入输出类型、特殊引脚和明显冲突项给出 warning，供工程师人工确认。当前该部分已经具备较稳定的单芯片辅助设计能力，是实现指标二的重要支撑。

### 4.4 测试代码生成
代码生成模块采用“模板骨架 + 企业知识 + RAG + 规则校验”的组合方案：

1. 生成测试计划
2. 组装代码骨架
3. 进行 RAG 增强和文本润色
4. 执行静态校验
5. 执行编译预检
6. 生成工程包

当前产出主要是 **工程初稿级代码**，适合测试工程师继续复核和完善。项目没有将这部分表述为“无需修改即可直接上机”，而是明确定位为辅助编程与初稿交付。

### 4.5 Agent Flow 与运行中心
本项目的核心升级之一是引入统一的 Agent Run 模型。当前已经实现：

- `full_ate_development`
- `post_review_delivery`
- `post_review_revision`

每次流程执行都会记录：

- `run_id`
- 步骤状态
- 开始/结束时间
- 质量分
- warning / error
- artifacts
- review 结果

前端工作台提供：

- 动态执行时间线
- `Thinking Feed`
- 工程复核面板
- 批准后交付文件展示

这使平台从“几个独立工具页”演进成了“统一可追踪的 Agent 化开发平台”。

### 4.6 工程师助手迁移
本项目后期将同伴开发的 `fa` 中有价值的能力迁移到了主框架，统一形成“工程师助手”模块。迁移内容包括：

- `workspace memory`
- 统一 `chat` 接口
- 本地知识目录 `data/knowledge`
- 多模态聊天接口预留
- OCR fallback 链路

迁移后的助手不再是独立平行系统，而是依附于主平台运行，能够读取：

- 最近 `run`
- 最近 `TestPlan`
- 最近资源映射结果
- 最近代码生成结果
- 最近诊断结果
- 本地知识片段

这使其从“普通聊天框”变成了带上下文的工程副驾。

### 4.7 轻量化桌面端与诊断能力
项目采用 `Electron + PyInstaller` 进行桌面端交付，用户无需单独安装 Python 即可运行。当前诊断部分实现了：

- 仿真 VI 波形展示
- 异常模式分析
- 故障类型提示
- 趋势可视化

这部分当前仍偏演示与验证性质，尚未接入真实工控机或 ATE 机台实时数据，因此在结题口径中保持了边界说明。

## 6. 对照课程指标的完成情况

### 6.1 指标一：Datasheet 复杂表格和图表识别准确率 ≥ 85%

- 参数准确率：`88.9%`
- 引脚准确率：`87.8%`
- 综合准确率：`88.6%`

结论：

- **已达到阶段性指标要求**

- 当前结果主要建立在文本型 PDF 与小样本测试集基础上
- 扫描件和复杂图像型页面仍是后续增强重点

### 6.2 指标二：DUT 原理图辅助设计建议准确率 ≥ 75%
已有阶段测试结果显示：

- 资源映射与辅助建议平均准确率：`84.1%`

结论：

- **已达到阶段性指标要求**

说明：

- 当前单芯片和常见场景表现较稳定
- 复杂多电源、多站点和更特殊的板级约束仍需要继续扩充规则库

### 6.3 指标三：测试程序辅助编程，自动生成代码符合测试要求，具备可实现性
已有样例验证显示：

- `5` 组样例中 `4` 组在少量人工修改后可作为工程初稿使用
- 当前代码生成模块已具备工程初稿级可实现性

结论：

- **基本达到指标要求**

说明：

- 当前更适合表述为“辅助编程与初稿生成”
- 尚不宜表述为“可完全无修改直接上机”

### 6.4 指标四：轻量化 AI 模型在不影响机台运行前提下完成实时诊断
已有阶段性验证数据表明：

- 空闲 CPU 占用约 `1%-3%`
- 诊断响应时间 `<1s`

结论：

- **已完成轻量化诊断演示验证**

说明：

- 因此这一条更适合表述为“阶段性完成演示验证”，而不是“最终闭环完全完成”

### 6.5 指标五：开发周期缩短 ≥ 50%，代码首检错误率降低 ≥ 70%

## 7. 创新点

本项目的主要创新点包括：

1. 将 `LLM + 工程规则 + ATE 平台知识` 融合，而不是纯大模型直出
2. 从 `Datasheet` 到 `TestPlan`、资源映射、代码初稿和交付结果形成统一辅助闭环
3. 引入统一的 `run / step / artifact` 模型，实现开发过程可追踪、可回看、可复核
4. 在主平台内统一迁入“工程师助手”，而不是维护第二套平行 AI 系统
5. 保留手动工具入口，同时叠加 Agent 工作台，实现“主流程自动化 + 单模块手动调试”的双模式协同

## 8. 存在的不足

当前项目仍存在以下不足：

- OCR 尚未成为全量主链，扫描版 PDF 适配能力仍有限
- 复杂图像型 pinout 和特殊封装图识别能力不足
- 代码生成仍以工程初稿为主，缺少更严格的真实 SDK 编译闭环
- 诊断模块尚未接入真实工控机数据
- 平台当前尚未实现批准后自动装载程序并驱动 ATE 真机测试
- 部分评分与解释机制仍可进一步优化，增强用户理解成本控制

## 9. 后续工作展望

后续优化方向包括：

1. 将 OCR fallback 深化为更完整的复杂页面处理链路
2. 增强图像型引脚识别与复杂表格识别能力
3. 扩展 `STS8200S` 平台规则与资源映射约束库
4. 建立更大规模的人工标注测试集，增强指标统计可信度
5. 接入更真实的编译验证和机台执行闭环
6. 将良率诊断从仿真演示扩展到真实工控机数据源
7. 完善工程师助手的多模态分析能力与协同解释能力

## 10. 结论

本项目已完成面向 ATE 测试开发场景的智能辅助平台主框架建设，并形成了从 `Datasheet` 提取到 `TestPlan`、资源映射、测试代码、工程复核和交付结果的核心链路。平台不仅完成了前后端一体化实现，还进一步通过 Agent 工作台、运行中心和工程师助手，把原本分散的单模块工具整合为统一的工程协同平台。

从课程指标角度看，项目在 `Datasheet` 识别准确率、辅助设计建议准确率、开发周期缩短和首检错误率下降等方面已经达到阶段性目标；在自动生成可实现测试代码方面基本达到要求；在轻量化实时诊断方面已完成演示验证，但真实工控机闭环仍待后续补充。

总体而言，本项目已经具备较强的阶段性成果价值和展示价值，并为后续向真实机台执行、客户级部署和更完整的多模态 ATE 工程助手演进打下了基础。
