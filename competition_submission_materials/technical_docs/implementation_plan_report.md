# 课题实施方案报告

## 1. 课题名称

基于语言大模型的智能 ATE 测试开发辅助平台

## 2. 课题背景与研究意义

集成电路测试开发是芯片从设计验证走向工程量产的重要环节。对于 ATE 测试工程师而言，测试开发通常需要从芯片 Datasheet 中人工整理电气参数、绝对最大额定值、推荐工作条件、引脚定义、功能表、时序要求和测试条件，再结合具体测试平台资源完成 TestPlan 编写、DUT 板卡资源分配、测试代码开发、程序调试和结果诊断。

传统开发模式主要依赖人工阅读 Datasheet 和工程经验判断，存在以下问题：

1. **资料解析耗时长**：Datasheet 中包含大量表格、图表、跨页说明和条件约束，人工整理容易遗漏。
2. **参数结构不统一**：不同厂商、不同类别芯片的参数命名、单位、表格格式和描述方式差异较大，难以直接复用。
3. **平台资源映射依赖经验**：DUT 引脚到 ATE 资源的映射需要综合考虑芯片类型、引脚方向、电压电流范围和测试平台资源约束，对初学者不友好。
4. **测试代码首版错误率较高**：测试程序开发涉及平台 API、通道号、测试项流程、量程设置和结果判定，人工初版代码容易出现低级错误。
5. **诊断与开发链路割裂**：测试开发、运行诊断、异常分析和经验复盘通常分散在不同工具中，缺少统一的智能化辅助平台。

随着语言大模型在文档理解、结构化信息抽取、代码生成和工程知识推理方面能力提升，将其引入 ATE 测试开发流程具有较高应用价值。本课题拟设计一套面向 STS8200S 测试平台的智能 ATE 测试开发辅助平台，形成从 Datasheet 到 TestPlan、资源映射、测试代码和诊断分析的半自动化工程流程，以提升测试开发效率、降低首版错误率，并为后续产品化应用奠定基础。

## 3. 总体建设目标

本课题的目标是建设一套面向芯片测试开发场景的智能辅助平台。最终系统应以 Datasheet 文档解析为入口，以 STS8200S 测试平台规则为约束，以 TestPlan、资源映射文件、测试代码和诊断结果为输出，形成可演示、可验证、可扩展的工程闭环。

最终方案不只实现基础演示功能，还应将准确率提升、速度优化、工程化部署、可维护性和后续产品化能力纳入整体设计。系统目标包括：

1. 支持不同芯片 Datasheet PDF 的上传、解析、有效页面筛选和结构化信息抽取。
2. 支持电气参数、测试条件、绝对最大额定值、推荐工作条件、时序参数、功能表和引脚定义的统一建模。
3. 支持生成标准化 TestPlan JSON 和 Excel 文件，便于测试工程师复核和二次编辑。
4. 支持结合 STS8200S 平台资源规则进行参数校验、资源适配和量程建议。
5. 支持根据芯片引脚和测试项生成 DUT 资源映射建议、PGS 配置、BOM 清单和辅助示意图。
6. 支持基于模板、RAG 知识增强和静态规则校验的 STS8200S 测试代码辅助生成。
7. 支持轻量化良率诊断和异常分析展示，并预留接入真实工控机数据的能力。
8. 支持 Windows 桌面端交付，用户无需安装 Python 即可运行系统。
9. 支持后续扩展为云端服务架构，实现 API Key 管理、任务队列、日志审计、用户权限和多人协作。

## 5. 系统总体架构方案

系统采用“前端交互层 + 后端服务层 + AI 能力层 + 工程规则层 + 文件输出层 + 桌面部署层”的分层架构。

### 5.1 前端交互层

前端负责用户操作、任务状态展示、结果预览和文件下载。目标页面包括：

- **仪表盘**：展示后端连接状态、当前处理任务、系统运行概况和诊断摘要。
- **提取器**：完成 PDF 上传、任务提交、进度展示、参数统计、引脚统计和 TestPlan 下载。
- **资源映射**：展示引脚到 STS8200S 资源的映射关系，生成 PGS、BOM 和辅助示意图。
- **代码实验室**：选择芯片类型、测试项和平台配置，生成 STS8200S 测试代码骨架。
- **故障诊断**：展示良率、异常事件、VI 波形、异常分类和诊断建议。

前端设计应兼顾比赛演示和工程使用，既能清晰展示 AI 处理结果，也能为测试工程师提供可复核、可下载、可编辑的交付物。

### 5.2 后端服务层

后端负责核心业务处理和文件生成，采用 REST API 方式对前端提供服务。主要模块包括：

- **testplan 模块**：处理 PDF 上传、文档解析、异步提取、任务状态查询和 TestPlan 下载。
- **resource-map 模块**：生成资源映射表、PinMapping 模板、PGS 配置、BOM 清单和 SVG 示意图。
- **codegen 模块**：根据测试项和平台规则生成测试代码，并执行静态规则检查。
- **rag 模块**：管理 STS8200S 平台知识片段，为代码生成和规则校验提供知识增强。
- **diagnosis 模块**：处理良率诊断、异常检测、波形数据和诊断结果输出。
- **runtime 模块**：管理桌面端启动、端口探测、后端健康检查和日志记录。

### 5.3 AI 能力层

AI 能力层是系统智能化的核心，主要包括：

1. **Datasheet 结构化抽取模型**：调用语言大模型识别 Datasheet 中的电气参数、测试条件、引脚定义、功能表和时序参数。
2. **RAG 知识增强机制**：将 STS8200S 平台规则、测试流程、API 说明和工程经验整理为可检索知识片段，为代码生成和规则校验提供依据。
3. **代码生成辅助模型**：结合模板、测试项定义和平台知识生成可复核的 C++ 测试代码骨架。
4. **异常检测模型**：用于诊断模块中的 VI 波形异常识别、良率趋势分析和故障类型归因。

AI 能力层不直接替代工程师决策，而是提供结构化初稿、工程建议和风险提示，最终结果仍需测试工程师复核。

### 5.4 工程规则层

工程规则层用于约束大模型输出，提升结果稳定性和可用性。规则来源包括：

- STS8200S 平台电压、电流、DIO、TMU、CBIT 等资源限制。
- 常见芯片引脚类型与资源需求规则。
- A/B/C 类参数分类规则。
- DC、AC、LDO、数字 IO、功能测试等测试项建模规则。
- 单位归一化、数值范围校验、缺失字段检查和异常值过滤规则。
- 测试代码结构、通道编号、结果上报和 API 调用约束。

通过“LLM 抽取 + 本地规则校验 + 人工复核”的组合，系统能够降低模型幻觉和不稳定输出对工程结果的影响。

### 5.5 文件输出层

系统最终输出应面向工程交付，包含：

- TestPlan Excel。
- TestPlan JSON。
- 资源映射 Excel。
- PGS 配置文件。
- BOM 清单。
- SVG 辅助示意图。
- C++ 测试代码骨架。
- 诊断结果 JSON。
- 运行日志和错误诊断信息。

输出文件应保持结构清晰、字段统一、可追溯、可复核，便于后续纳入真实测试开发流程。

### 5.6 桌面部署层

系统以 Windows 桌面应用作为主要交付形态。目标方案为 Electron 前端内置 FastAPI 后端运行时，并通过 PyInstaller 将 Python 后端打包为独立可执行文件。用户安装后可直接启动应用，无需额外安装 Python 环境。

桌面端需要具备：

- 自动启动本地后端服务。
- 自动选择可用端口。
- 将真实 API 地址传递给前端。
- 提供健康检查与错误提示。
- 记录后端启动日志和运行日志。
- 支持后续安装包签名、版本更新和用户配置管理。

## 6. 关键技术路线

### 6.1 Datasheet PDF 解析与有效页面筛选

系统首先对 PDF 执行文本和表格解析，并将页面内容转换为可供模型处理的结构化上下文。由于 Datasheet 中并非所有页面都与测试开发相关，系统需要设计有效页面筛选机制，优先保留以下内容：

- Electrical Characteristics。
- Absolute Maximum Ratings。
- Recommended Operating Conditions。
- Pin Configuration / Pin Description。
- Timing Characteristics。
- Truth Table / Function Table。
- Package Information 中与引脚相关的页面。

对封面、目录、修订历史、订购信息、免责声明等低价值页面，系统根据关键词、文本密度、表格密度和页面结构特征进行过滤。该策略可以减少无效上下文输入，降低大模型调用成本，并提升提取速度。

对于扫描型 PDF 或图像型 pinout，目标方案预留 OCR 和视觉模型接口，用于补充纯文本解析无法覆盖的内容。

### 6.2 大模型结构化信息抽取

对筛选后的有效页面，系统按照页面内容和章节类型进行分块，并调用语言大模型执行结构化抽取。提示词需要明确约束输出 JSON 结构，字段包括：

- 参数名称。
- 参数符号。
- 最小值、典型值、最大值。
- 单位。
- 测试条件。
- 参数类别。
- 测试场景。
- 所属章节。
- 引脚编号、引脚名称、方向和功能说明。
- 模型置信度和来源页面。

模型输出后，后端需要执行 JSON 解析、字段标准化、单位归一化、重复项合并、异常值过滤和缺失字段标记。对于不确定结果，应保留置信度和来源页面，方便工程师复核。

### 6.3 本地规则兜底与质量控制

大模型对复杂 PDF 的解析存在不稳定性，因此目标方案需要引入本地规则兜底机制：

- 对常见电气参数进行正则提取补充。
- 对常见芯片类型进行场景识别。
- 对常见引脚命名进行方向推断。
- 对单位和数值范围进行标准化处理。
- 对缺少测试条件、缺少单位或明显越界的参数进行风险标记。
- 对模型无法识别的引脚表提供人工 PinMapping 模板补录入口。

系统最终应形成“模型主抽取、规则补充、平台约束校验、人工复核”的质量控制闭环。

### 6.4 STS8200S 平台兼容性校验

平台校验模块根据 STS8200S 的资源能力和约束，对提取结果进行工程化检查：

- 电压范围是否超出平台可支持范围。
- 电流范围是否需要特殊量程或保护策略。
- DIO 通道数量是否满足芯片引脚需求。
- TMU、CBIT、VI、FH/SH 等资源是否适配目标测试项。
- 测试条件是否完整。
- 参数分类是否符合 TestPlan 编写习惯。
- 是否存在需要人工确认的高风险测试项。

校验结果应写入 TestPlan，并在前端以告警或建议形式展示。

### 6.5 TestPlan 自动生成

TestPlan 生成模块将模型抽取结果、规则校验结果和平台建议统一转换为工程文件。目标输出包括 JSON 和 Excel 两类格式。

JSON 用于系统内部流转和模块间复用，Excel 用于测试工程师复核、修改和提交。TestPlan 中应包含：

- 芯片基础信息。
- 参数列表。
- A/B/C 类参数分类。
- DC/AC/功能/电源类测试项。
- 引脚定义。
- 测试条件。
- STS8200S 兼容性校验结果。
- 风险提示和人工确认项。

### 6.6 DUT 资源映射与辅助设计

资源映射模块以 TestPlan 和引脚定义为输入，生成面向 STS8200S 的资源分配建议。基本流程为：

1. 读取 TestPlan JSON。
2. 获取芯片类型、引脚定义和测试项列表。
3. 按输入、输出、电源、地、NC、双向 IO 等类别进行引脚分组。
4. 根据测试项需求分配 DIO、VI、TMU、CBIT 等资源。
5. 检查资源冲突、通道不足和特殊量程需求。
6. 输出资源映射表、PGS 配置、BOM 清单和 SVG 辅助示意图。

目标方案中，资源映射不应只给出静态结果，还应给出推荐原因、风险项和人工可调整入口，为 DUT 原理图设计提供可解释的辅助建议。

### 6.7 测试代码辅助生成

测试代码生成采用“模板化生成 + RAG 知识增强 + 静态规则校验”的技术路线。

模板化生成用于保证测试函数结构、初始化流程、结果上报和基础注释稳定；RAG 用于检索 STS8200S API、平台约束和典型测试流程；大模型用于根据 TestPlan 补充参数化测试逻辑和代码说明；静态校验器用于检查通道编号、测试函数结构、参数范围和常见 API 风险。

生成代码的目标定位是“可实现的工程初稿”，而不是直接替代人工上线代码。系统应在输出中明确标记需要人工确认的位置，例如真实通道号、特殊测试条件、保护电路配置和平台 API 差异。

### 6.8 轻量化良率诊断

诊断模块面向工控机侧轻量化运行场景，目标是辅助测试工程师快速识别良率异常和可能原因。方案包括：

- 采集或模拟 VI 波形、良率、Bin 分布、FTY 和异常事件。
- 对波形异常、良率突降、特定 Bin 增长等情况进行检测。
- 输出故障分类、可能原因和处理建议。
- 前端展示趋势图、波形图、异常列表和诊断摘要。

比赛阶段可使用仿真数据验证诊断流程；最终产品化方向应支持接入真实工控机数据，并保证诊断模块低资源占用，不影响机台运行。

## 7. 系统实现方案

### 7.1 后端实现方案

后端采用 FastAPI 构建服务接口，设计原则包括：

- 上传和提取任务异步执行，避免大文件处理阻塞前端。
- 任务状态可查询，便于前端实时展示进度。
- 文件存储分区管理，区分上传文件、处理结果、缓存文件和日志文件。
- 统一响应格式，便于前端处理正常结果和错误结果。
- 记录关键运行日志，支持定位其他电脑上的安装和运行问题。
- 支持缓存机制，避免同一 PDF 重复调用模型。
- 提供健康检查接口，供桌面端判断后端是否启动成功。

### 7.2 前端实现方案

前端采用 React、TypeScript 和 Tailwind CSS 进行组件化开发。实现原则包括：

- 上传、提取、轮询、完成和错误状态统一管理。
- 关键结果以参数统计、引脚列表、告警信息和文件下载入口形式展示。
- 多模块共享 `file_id` 或任务结果，实现提取器、资源映射和代码生成之间的数据流转。
- 在上传前执行后端 API 预检查，避免后端未连接时用户长时间等待。
- 对错误信息进行可读化展示，降低非开发人员排查难度。

### 7.3 桌面端实现方案

桌面端基于 Electron 实现，后端通过 PyInstaller 打包为 `backend-server.exe`。启动流程为：

1. 用户启动桌面应用。
2. Electron 主进程检测可用端口。
3. Electron 启动内置后端可执行文件。
4. Electron 轮询后端健康检查接口。
5. 后端启动成功后，将 API 地址注入前端页面。
6. 前端使用注入地址访问后端 API。
7. 后端日志写入本地文件，便于排查安装和运行问题。

该方案适合比赛演示和小范围试用。面向公开客户使用时，可进一步演进为“桌面客户端 + 云端后端”的架构。

### 7.4 云端产品化扩展方案

为了满足后续多人使用和客户级交付，系统预留云端部署扩展方向：

- 大模型 API Key 从客户端迁移到云端统一管理，避免密钥泄露。
- 后端服务部署到云服务器，统一处理 PDF 解析、模型调用和文件生成。
- 引入用户账号、权限管理和任务历史。
- 引入任务队列，支持并发处理多个 Datasheet。
- 引入日志审计和错误上报，便于运维。
- 支持客户端自动更新和版本管理。
- 支持不同客户的数据隔离和私有化部署。

桌面端本地部署适合作为比赛与离线演示版本，云端部署适合作为后续商业化和客户级使用版本。

## 9. 测试验证方案

### 9.1 Datasheet 识别准确率验证

选择多份不同类型 Datasheet，人工标注电气参数、引脚定义、工作条件和关键表格内容作为标准答案。将系统输出与标准答案逐项比对，统计准确率、召回率和错误类型。

重点验证：

- 参数名称识别准确率。
- 参数数值和单位识别准确率。
- 测试条件识别准确率。
- 引脚编号和引脚功能识别准确率。
- 复杂表格和跨页表格处理能力。

### 9.2 DUT 辅助设计建议验证

针对典型芯片建立人工资源映射方案，将系统生成的资源映射、PGS 配置和辅助示意图与人工方案对比，统计建议准确率和人工可接受率。

重点验证：

- 引脚方向判断是否正确。
- 电源、地、输入、输出、双向 IO 资源分配是否合理。
- 是否存在资源冲突。
- 是否能提示特殊量程和风险项。

### 9.3 测试代码可实现性验证

选择典型测试项生成 C++ 测试代码，由工程人员进行审查或静态检查。统计代码结构完整性、API 使用合理性、测试逻辑可实现性和人工修改量。

重点验证：

- 代码结构是否符合 STS8200S 测试开发习惯。
- 测试项参数是否来自 TestPlan。
- 是否存在明显通道、量程或 API 错误。
- 人工修改成本是否低于传统从零编写方式。

### 9.4 轻量化运行验证

在目标电脑或工控机环境中运行桌面应用，记录启动时间、后端启动时间、内存占用、CPU 占用、任务响应时间和诊断模块运行开销，验证系统是否适合轻量化部署。

### 9.5 开发效率对比验证

设计传统人工流程与 AI 辅助流程对比实验。对同一份 Datasheet，分别统计人工整理 TestPlan、生成资源映射和编写首版测试代码的耗时，并记录首检错误数量。通过对比验证开发周期缩短和错误率降低效果。

## 10. 风险分析与应对措施

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 大模型输出不稳定 | 影响参数提取准确率 | 使用 JSON 结构约束、结果清洗、本地规则兜底和人工复核机制 |
| 图像型 pinout 难以识别 | 影响引脚提取完整性 | 引入 OCR/视觉模型接口，并提供 PinMapping 人工补录模板 |
| 不同厂商 Datasheet 格式差异大 | 影响泛化能力 | 建立多芯片测试集，持续补充页面过滤规则和提示词策略 |
| 大模型接口延迟较高 | 影响整体提取速度 | 使用有效页面过滤、缓存、并发调度和局部规则提取降低调用量 |
| 桌面端安装包体积较大 | 影响分发体验 | 优化 PyInstaller 打包内容，区分比赛版、轻量版和云端版 |
| 客户端 API Key 暴露 | 不适合公开分发 | 产品化阶段改为用户自配 Key 或云端统一代理调用 |
| 真实机台数据不足 | 影响诊断模块真实性 | 比赛阶段使用仿真数据验证流程，后续接入工控机真实数据 |
| 生成代码直接上机存在风险 | 可能影响测试安全 | 输出定位为工程初稿，增加静态校验、风险标记和人工确认机制 |
