## 3. 当前已具备的系统能力

当前项目已经具备以下能力：

1. 用户可通过桌面端上传芯片 Datasheet PDF。
2. 系统可自动解析有效页面，并调用大语言模型提取测试参数。
3. 系统可对低价值页面进行过滤，减少无效模型调用，提高提取速度。
4. 系统可生成结构化 TestPlan JSON 和 Excel 文件。
5. 系统可基于 STS8200S 平台规则进行基础兼容性校验和量程建议。
6. 系统可基于引脚定义生成资源映射、PGS 配置、BOM 和 SVG 辅助示意图。
7. 系统可根据芯片类型和测试项生成 STS8200S C++ 测试代码骨架。
8. 系统提供基于仿真 VI 波形的轻量化良率诊断演示。
9. 系统已完成 Windows 桌面安装包构建，用户无需安装 Python 即可运行。

从项目目标来看，当前已完成核心功能开发和工程化打包，后续工作重点将从“功能实现”转向“指标量化验证、测试数据补充、复杂场景增强、演示视频录制和结题材料完善”。

## 5. 已完成工作详情

### 5.1 系统架构与工程框架搭建

项目采用前后端分离架构，并进一步封装为 Windows 桌面应用。前端负责用户界面、任务状态管理和结果展示，后端负责 PDF 解析、模型调用、规则校验、资源映射、代码生成和诊断分析。

已完成内容包括：

- 搭建 React + TypeScript + Vite 前端工程。
- 搭建 FastAPI 后端工程。
- 设计统一 API 响应结构。
- 实现前端 API 客户端封装。
- 实现 Electron 桌面端主进程。
- 实现 PyInstaller 后端打包入口。
- 实现桌面端自动启动内置 FastAPI 后端。
- 增加后端启动日志，便于在其他电脑上排查启动问题。

该部分完成度评估为 **100%**。它解决了系统从 Web 原型向桌面端工程应用转化的问题，使项目具备较完整的交付形态。

### 5.2 Datasheet PDF 上传与解析模块

该模块是系统的第一入口，负责接收芯片数据手册并解析其中有效内容。

已完成内容包括：

- PDF 文件上传接口。
- 文件类型校验，仅允许上传 PDF。
- 文件大小限制，避免超大文件阻塞服务。
- 上传文件唯一 ID 生成。
- PDF 文本内容提取。
- 表格内容融合到页面文本。
- 上传目录和处理目录自动创建。
- 异步提取任务接口和任务状态查询接口。

该部分完成度评估为 **90%**。当前系统已经能够稳定接收 Datasheet，并将其传递给后续提取流程。剩余优化点主要是扫描型 PDF、图像型页面和复杂跨页表格解析。

### 5.3 页面过滤与提取性能优化

Datasheet 中包含大量封面、目录、修订记录、订购信息等低价值页面。若全部提交给大模型，会增加接口耗时和费用，并可能降低提取质量。因此项目中期阶段重点优化了页面过滤和缓存机制。

已完成内容包括：

- 根据关键词识别关键页面，例如 Electrical Characteristics、Absolute Maximum Ratings、Recommended Operating Conditions、Pin Description 等。
- 跳过封面、目录、修订记录和空白页等明显无效页面。
- 对页面文本进行紧凑化处理，减少无效上下文。
- 对同一 PDF 和页码范围生成缓存键。
- 命中缓存时直接返回历史 TestPlan 结果。
- 增加并发提取和接口超时控制。

该部分完成度评估为 **90%**。优化后，重复提取和无效页较多的文件处理速度明显提升，系统可用性增强。后续可继续根据测试集统计进一步调优页面分类规则。

### 5.4 大语言模型结构化提取模块

该模块是项目的核心能力之一，用于将 Datasheet 中半结构化文本转换为可计算、可校验的结构化数据。

已完成内容包括：

- 接入 DeepSeek API。
- 设计结构化提取 Prompt。
- 约束模型输出 JSON 格式。
- 提取电气参数、测试条件、单位和参数类别。
- 提取引脚编号、引脚名称、方向和功能说明。
- 对模型输出进行 JSON 解析、字段标准化和错误处理。
- 对提取结果进行去重、合并和分类。

该部分完成度评估为 **85%**。当前系统已支持常见数字芯片、LDO、EEPROM 和通用芯片的参数提取场景。对于文字型表格，提取效果较好；对于扫描图像或复杂图形，仍需进一步优化。

### 5.5 本地规则兜底与质量控制

为了避免完全依赖大模型带来的不稳定性，项目在中期阶段加入本地规则补充机制。

已完成内容包括：

- 对绝对最大额定值进行规则提取。
- 对推荐工作条件进行规则提取。
- 对部分时序参数进行本地识别。
- 对部分常见视觉 pinout 增加兜底。
- 对模型输出中的无效字段进行过滤。
- 对重复参数进行合并。
- 对缺失测试条件生成提醒。

该部分完成度评估为 **80%**。该设计使系统形成“LLM 提取 + 本地规则补充 + 平台规则校验”的混合方案，降低了单一模型输出不稳定对最终结果的影响。

### 5.6 TestPlan 自动生成模块

TestPlan 模块负责将参数提取结果转换为测试工程师可复核和复用的文件。

已完成内容包括：

- 生成 TestPlan JSON。
- 生成 TestPlan Excel。
- 按 A 类必测参数、B 类推荐参数、C 类可选参数进行分类。
- 统计 DC、AC、LDO 等测试项数量。
- 输出 STS8200S 兼容性结果。
- 输出量程推荐和风险提示。
- 前端展示参数统计、置信度、引脚数量和下载入口。

该部分完成度评估为 **95%**，是当前完成度最高的核心模块之一。剩余工作主要是进一步优化 Excel 样式和字段说明，使其更接近真实工程交付格式。

### 5.7 引脚定义识别模块

引脚定义是资源映射和代码生成的基础。中期阶段系统已实现文字型引脚表的自动提取，并在前端进行表格展示。

已完成内容包括：

- 从 Datasheet 文本中识别 pin table。
- 提取 pin number、pin name、function、direction 等字段。
- 根据引脚名称推断输入、输出、电源、地和 NC 类型。
- 前端展示引脚定义表。
- 模块二资源映射可自动读取模块一提取的引脚定义。
- 对部分图像型 pinout 增加已知规则补充。

该部分完成度评估为 **75%**。当前不足是图像型 pinout、扫描版 Datasheet 和复杂封装图的自动识别能力仍有限，后续计划接入 OCR 或视觉模型。

## 8. 当前存在的问题

### 8.1 图像型 Datasheet 内容识别不足

当前系统对文本型 PDF 效果较好，但对于扫描版 PDF、图片型 pinout、复杂波形图和跨页表格识别能力仍有限。这会影响部分芯片的引脚提取完整度。

### 8.2 资源映射规则仍需扩充

资源映射模块已能处理基础数字芯片、LDO 等典型场景，但对于复杂多电源、多站点、多模式测试场景仍需更完整的平台规则支持。

### 8.3 测试代码生成仍需人工复核

当前生成代码可以作为工程初稿，但尚未接入真实 STS8200S SDK 编译链路，无法完全自动证明代码可上机执行。

### 8.4 良率诊断仍处于演示阶段

诊断模块目前使用仿真波形数据，能够展示算法思路，但尚未接入真实工控机或 ATE 运行数据。

### 8.5 客户级部署能力不足

当前安装包适合比赛演示和小范围试用，但若面向客户长期使用，需要云端后端、账号体系、日志上报、API Key 安全管理和软件签名。

## 9. 已采取的改进措施

针对中期开发过程中发现的问题，项目已完成以下改进：

1. **针对提取速度慢的问题**：增加页面过滤、缓存机制、并发提取和本地规则补充。
2. **针对模型输出不稳定的问题**：加强 JSON 输出约束、字段标准化和异常值过滤。
3. **针对引脚提取不完整的问题**：增加文字型引脚表提取和部分图像 pinout 兜底规则。
4. **针对桌面端依赖 Python 的问题**：使用 PyInstaller 打包后端运行时。
5. **针对安装包过大的问题**：优化 PyInstaller 排除项，显著减少无关依赖。
6. **针对其他电脑后端无法定位的问题**：增加 `backend-launch.log` 启动诊断日志。
7. **针对上传卡住的问题**：前端上传前增加后端 API 健康检查和请求超时。
