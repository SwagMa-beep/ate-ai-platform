# 结题技术报告大纲

> 本文档用于最终结题技术报告扩写。详细实现细节可引用 `technical_docs/technical_implementation_details.md`，系统架构图可引用 `diagrams/system_architecture.md`。

## 摘要

本项目面向集成电路 ATE 测试开发场景，设计并实现了一套基于语言大模型的智能 ATE 测试开发辅助平台。系统以芯片 Datasheet PDF 为输入，结合 PDF 解析、页面过滤、DeepSeek 大模型结构化提取、本地规则兜底和 STS8200S 平台规则校验，自动生成 TestPlan、资源映射、PGS、BOM、SVG 辅助示意图和 C++ 测试代码骨架，并提供轻量化良率诊断展示。项目已完成 Windows 桌面端打包，用户无需安装 Python 即可运行。

## 1. 项目背景与意义

### 1.1 ATE 测试开发流程现状

传统 ATE 测试开发通常包括 Datasheet 阅读、参数整理、TestPlan 编写、DUT 资源映射、测试代码开发、调试验证和良率诊断等环节。该流程依赖测试工程师经验，人工整理耗时长，首版代码容易出现通道配置、量程设置和测试条件遗漏等问题。

### 1.2 主要痛点

- Datasheet 中复杂表格、图形和跨页说明较多，人工提取效率低。
- 不同厂商 Datasheet 格式差异大，参数字段难以统一。
- DUT 引脚到 ATE 资源的映射依赖工程经验。
- 测试程序初稿编写重复性强，容易产生低级错误。
- 测试开发和良率诊断工具割裂，缺少统一辅助平台。

### 1.3 项目意义

本项目将语言大模型的文档理解能力与 ATE 工程规则结合，用于提升测试开发效率、降低初版错误率，并为后续真实客户级产品化提供基础。

## 2. 赛题指标与项目目标

| 赛题指标 | 项目目标 |
|---|---|
| Datasheet 复杂表格和图表识别准确率 ≥85% | 构建 PDF 解析、页面过滤、LLM 提取、本地规则兜底和结果标准化流程 |
| DUT 原理图辅助设计建议准确率 ≥75% | 根据引脚定义和 STS8200S 资源规则生成资源映射、PGS、BOM 和 SVG |
| 自动生成测试程序，具备可实现性 | 基于模板、RAG 和静态校验生成 C++ 测试代码骨架 |
| 工控机轻量化运行并完成实时诊断 | 通过 Electron 桌面端和本地后端实现轻量化诊断展示 |
| 开发周期缩短 ≥50%，首检错误率降低 ≥70% | 自动生成 TestPlan、资源映射和代码初稿，减少人工重复工作 |

## 3. 系统总体架构

系统采用“前端交互层 + 后端服务层 + AI 能力层 + 工程规则层 + 文件输出层 + 桌面部署层”的分层架构。

### 3.1 前端交互层

前端基于 React、TypeScript、Vite 和 Electron 实现，主要页面包括仪表盘、Datasheet 提取器、资源映射、代码实验室和故障诊断页面。

### 3.2 后端服务层

后端基于 FastAPI 实现，正式挂载的接口包括：

- `/api/v1/testplan/*`
- `/api/v1/resource-map/*`
- `/api/v1/codegen/*`
- `/api/v1/rag/*`
- `/api/v1/diagnosis/*`

### 3.3 AI 与工程规则层

AI 层调用 DeepSeek API 完成结构化提取和代码辅助生成；工程规则层负责本地规则兜底、STS8200S 兼容性校验和代码静态检查。

### 3.4 桌面部署层

桌面端使用 Electron 封装前端，后端通过 PyInstaller 打包为 `backend-server.exe`。Electron 主进程负责自动选择端口、启动后端、轮询 `/health` 并写入 `backend-launch.log`。

## 4. 关键技术实现

### 4.1 Datasheet PDF 解析与页面过滤

系统通过 PDF 解析器读取每页文本和表格内容，并将页面转换为统一 chunk。随后在 `TestPlanService._filter_and_batch_chunks()` 中对页面进行打分，优先保留 Electrical Characteristics、Absolute Maximum Ratings、Recommended Operating Conditions、Pin Description、Timing Characteristics 等页面，过滤封面、目录、修订历史、订购信息和空白页。

页面过滤后，系统会对文本进行紧凑化处理，减少重复表格内容和无效空白，再将有效页面分批提交给大模型。该策略可以减少 DeepSeek API 调用量，提高提取速度。

### 4.2 大模型结构化提取

大模型提取由 `backend/app/services/llm_extractor.py` 实现。系统通过 Prompt 约束 DeepSeek 输出 JSON，字段包括参数名称、符号、最小值、典型值、最大值、单位、测试条件、参数类别、引脚编号、引脚名称、方向和功能说明。

由于模型可能返回 Markdown 代码块或额外解释文本，系统实现 `_extract_json_text()` 从响应中提取 JSON，再进行字段标准化、单位归一化、去重和异常处理。

### 4.3 并发提取与缓存

系统使用 `ThreadPoolExecutor` 并发处理多个页面 chunk，接口参数 `max_workers` 控制并发数。为了提高重复测试速度，系统构建缓存键，缓存键由缓存版本、PDF 内容和页码范围共同生成。命中缓存时，直接返回历史 Excel/JSON 结果，跳过 PDF 解析和模型调用。

### 4.4 本地规则兜底

系统在大模型之外补充本地规则，用于提取常见 Absolute Maximum Ratings、Recommended Operating Conditions、文本型 pin table 和部分常见 pinout。若本地已经稳定提取出引脚表，则减少对应页面的 LLM 调用。

### 4.5 STS8200S 平台规则校验

平台校验模块对提取参数进行电压范围、电流范围、DIO 通道、CBIT/TMU/VI 资源、缺失单位、缺失测试条件和异常值检查。校验结果写入 TestPlan，并在前端以 warning 和量程建议形式展示。

### 4.6 TestPlan 自动生成

系统输出 TestPlan JSON 和 Excel。JSON 用于后续资源映射和代码生成模块继续读取，Excel 用于工程师复核和提交。TestPlan 中包含芯片信息、参数列表、A/B/C 类参数、DC/AC/LDO 测试项、引脚定义、STS8200S 校验结果和风险提示。

### 4.7 DUT 资源映射与辅助设计

资源映射模块由 `resource_mapping_service.py` 实现。模块读取 TestPlan 和引脚定义，根据输入、输出、电源、地、NC、双向 IO 等类型进行分组，并根据测试项需求分配 STS8200S 资源。输出包括资源映射 Excel、PGS 配置、BOM 清单和 SVG 辅助示意图。

### 4.8 测试代码辅助生成

当前正式代码生成接口为 `/api/v1/codegen/*`。系统基于模板生成 C++ 测试代码骨架，并结合 RAG 检索 STS8200S 知识片段进行增强。生成后使用 `CodeValidator` 做静态检查，返回代码文本、行数、函数数和风险提示。

### 4.9 轻量化良率诊断

诊断模块使用仿真 VI 波形和异常检测逻辑，输出良率、FTY、Bin 分布、异常事件和故障类型。当前主要用于验证轻量化诊断流程，后续可接入真实工控机数据。

## 7. 工程化实现

### 7.1 后端工程化

后端使用 FastAPI，采用模块化路由和服务层设计。统一响应结构便于前端解析，异步任务机制便于处理大文件，日志机制便于排查提取、模型调用和文件生成问题。

### 7.2 前端工程化

前端使用 React 组件化实现，`backend.ts` 统一封装 API 请求，`extractionStore.ts` 管理提取状态和跨页面数据共享。

### 7.3 桌面端工程化

Electron 主进程负责后端运行时管理。打包环境中启动 `backend-server.exe`，开发环境中启动 Python 后端。系统自动探测端口并轮询 `/health`，保证前端只在后端可用后调用业务接口。

## 8. 创新点

1. 将 LLM 文档理解与 ATE 工程规则结合，降低单纯模型输出不稳定的风险。
2. 从 Datasheet 到 TestPlan、资源映射和代码生成形成完整辅助闭环。
3. 面向 STS8200S 平台进行资源约束和量程建议。
4. 通过 Electron + PyInstaller 实现无需 Python 的桌面端交付。
5. 将测试开发辅助和良率诊断展示放入同一平台，为后续真实机台数据接入预留空间。

## 9. 不足与改进方向

- 图像型 pinout 和扫描 PDF 识别能力仍需增强。
- 复杂 DUT 板卡约束和多站点测试规则仍需扩展。
- 测试代码生成尚未接入真实 STS8200S SDK 编译链路。
- 良率诊断暂未接入真实工控机数据。
- 客户级使用需要云端后端、账号体系、任务队列、日志审计和 API Key 安全管理。

## 10. 总结

本项目已经完成智能 ATE 测试开发辅助平台的核心闭环，实现了 Datasheet 解析、TestPlan 生成、平台规则校验、资源映射、测试代码生成、良率诊断和桌面端打包。中期测试结果表明，系统在主要赛题指标上达到阶段性目标，后续可继续向真实机台接入和客户级产品化方向演进。
